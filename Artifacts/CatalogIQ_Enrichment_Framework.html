A systematic methodology for building proprietary enrichment logic across B2B verticals — promotional products, pet supply, and ecommerce. How to construct it, compound it, and defend it as IP.

Version 1.0 · May 2026 · Confidential
01 · What Framework We Are Actually Using

We did not start with a framework. We started with a question: why does Kodiak Wholesale appear in ChatGPT when zero PPAI distributors do? The framework emerged from trying to answer that question systematically. It has a name now. We call it the STRIDE Enrichment Framework.

STRIDE stands for: Signal Audit → Taxonomy Mapping → Raw Data Acquisition → Intent Enrichment → Distribution → Evidence Loop. Each step builds on the last. The evidence loop at the end feeds back into the signal audit at the beginning. That feedback loop is how the IP compounds over time.

S
Signal Audit
Run structured buyer-intent queries. Record who appears, what fields get cited, what language drives citations. This is your measurement layer.
T
Taxonomy Mapping
Map the vertical's data standards to ACP/GMC/UCP field schemas. Identify what exists in source data vs what must be generated.
R
Raw Data Acquisition
Pull from the vertical's authoritative data source. For promo: PromoStandards. For pet: NAPPA/distributor APIs. For ecommerce: product feeds.
I
Intent Enrichment
Generate the fields that do not exist in source data. Use-case tags, buyer-intent descriptions, FAQ pairs, procurement language. This is the LLM layer.
D
Distribution
Format enriched data as ACP feed (ChatGPT), GMC feed (Google AI Mode), UCP manifest (Universal Commerce Protocol). Push to endpoints.
E
Evidence Loop
Re-run signal audit. Measure citation rate change. Record which fields drove lift. Feed findings back into enrichment logic. This is how IP compounds.
↻
The Compounding Loop
Each Evidence Loop cycle produces a finding. Each finding improves enrichment accuracy. Across 20 customers and 3 verticals, these findings become a proprietary dataset that no competitor can replicate without running the same experiment for the same duration. This is the IP moat.
The Core Insight The framework we are using is fundamentally a measurement-first enrichment system. Traditional SEO tools enrich first and measure later. STRIDE measures first so that enrichment is targeted at the specific fields that drive AI citation lift — not at all possible fields. This is why production time drove +18pp citation lift in our drinkware experiment while MOQ declined. The measurement told us which field to prioritize before we enriched.

Every product record — regardless of vertical — needs to be assessed against seven layers. The layers are universal. The specific fields within each layer are vertical-specific. This is the structural insight that makes the framework scalable across verticals.

The Critical Distinction Layers 1-3 are infrastructure — without them, the feed is rejected. Layers 4-6 are quality — they improve ranking within the feed. Layer 7 is the moat — no source data contains these fields. They must be generated by your enrichment pipeline and they are what drives semantic query matching. Every competitor who builds a feed tool can do Layers 1-6. Only you can do Layer 7 correctly for each vertical because Layer 7 requires vertical-specific domain knowledge about how buyers in each industry actually phrase their queries.

04 · The Enrichment Pipeline Architecture

Every record passing through the pipeline goes through seven steps in sequence. The steps are the same across verticals. The specific transformation logic at each step is vertical-specific. This is where the IP lives.

1
Ingest — Pull from authoritative source
Connect to the vertical's primary data source. For promo: PSRESTful API (PDS + PPC + INV + MCS endpoints). For pet: distributor-specific APIs or feed files. For ecommerce: platform API or CSV. Store raw response as immutable source-of-truth record.
IP component: connector library per vertical + authentication management
2
Normalize — Resolve vertical-specific quirks
Map source field names to canonical schema. Standardize color names (SanMar "Coastal Blue" → "Navy"). Resolve size codes. Handle missing fields. Flatten variant arrays into group_id structure. This normalization library is built from real supplier data and takes months to make production-quality.
IP component: normalization rules library — grows with each supplier processed
3
Score — Assess completeness against ACP requirements
Run each record against the 12-field ACP readiness rubric. Score 0-100. Flag critical gaps (missing price, missing image, missing required identity fields). Flag high-value gaps (missing production time, missing decoration method). Prioritize enrichment effort by revenue-weighted field impact score.
IP component: vertical-specific scoring rubric + revenue-weighting model
4
Enrich — Generate AI-native fields via LLM
This is the core IP step. Use Claude API to generate: ACP-optimized title, procurement-intent description, 5-8 FAQ pairs, use-case tags, buyer-intent description. The prompt engineering for each vertical is built from empirical data about which query types and which language patterns drive AI citations. Each Evidence Loop cycle improves the prompts.
IP component: vertical-specific prompt library — the primary defensible asset
5
Validate — Ensure feed compliance
Check all required fields present and valid. Verify price is numeric. Confirm image URLs resolve. Validate Schema.org property names. Check group_id consistency across variants. Flag identifier_exists=false for products without GTIN. Run the equivalent of Google Rich Results Test programmatically.
IP component: validation ruleset + error classification system
6
Format + Distribute — Push to AI commerce endpoints
Serialize to JSONL (ACP), XML (GMC), and UCP manifest. Push ACP feed via SFTP to OpenAI endpoint (up to every 15 minutes for live pricing). Push GMC feed to Google Merchant Center. Publish UCP manifest at /.well-known/ucp. Track submission confirmations and feed health status.
IP component: multi-format serializer + endpoint management per customer
7
Measure — Run Evidence Loop
Run the same 20 buyer-intent queries that were run at baseline. Record result type per query per platform. Calculate citation rate change. Identify which specific fields correlated with citation lift. Feed findings back into Step 4 prompt library. Aggregate anonymized findings across all customers.
IP component: query taxonomy library + citation tracking database + cross-customer aggregation

06 · How the IP Compounds Over Time

The framework produces three categories of IP. Each category compounds differently. Understanding this is critical for prioritizing what to build and when.

Category 1 — Normalization Knowledge
Supplier-specific quirk library
SanMar uses "Coastal Blue" for Navy. PCNA uses non-standard size codes. HIT has incomplete decorationArray data. These quirks take months to discover and map. Every supplier processed adds to the library.
How it compounds
Linear with number of suppliers processed. After 50 suppliers, normalization quality is dramatically better than a new entrant starting from zero. After 200, it is practically unreplicable in under 12 months.
Build rate
Grows with each customer catalog processed. 10 customers across 30 suppliers = 30 supplier quirk profiles. Passive accumulation.
Category 2 — Prompt Engineering Library
Vertical-specific enrichment prompts
The Claude API prompts that generate use-case tags, buyer-intent descriptions, and FAQ pairs for each vertical. Each prompt iteration is informed by Evidence Loop data showing which language patterns drive citation lift.
How it compounds
Exponential with measurement cycles. Each 14-day evidence loop improves prompts. Better prompts produce better enrichment. Better enrichment produces more citations. More citations produce better measurement data. Classic compounding loop.
Build rate
One measurement cycle every 14 days = 26 improvement cycles per year. After 6 months you have prompts that have been iterated 13 times. A new entrant starts at iteration 0.
Category 3 — Citation Dataset
Cross-customer AI citation database
Anonymized data showing which fields drove citation lift for which query types across which platforms. This dataset answers questions nobody in the industry has answered: which PromoStandards fields predict ChatGPT citation rate?
How it compounds
Network effects. 1 customer = 60 data points per cycle. 10 customers = 600 per cycle. 50 customers = 3,000 per cycle. Statistical significance improves with scale. Findings become publishable research that attracts more customers.
Build rate
Grows with customer count and measurement frequency. 5 customers running bi-weekly measurements = 600 citation data points per month accumulating continuously.

08 · The Query Taxonomy — The Most Underbuilt Asset

The single highest-leverage IP asset we are building — and the one that is most underdeveloped — is the query taxonomy. Every citation data point we collect is only as valuable as the query taxonomy it is organized around.

We currently have 8 query types from the drinkware experiment: specification, occasion, comparison, budget, conversational, multi-objective, negative constraint, workflow. These 8 types were derived from intuition. They need to be derived from data.

09 · Improving the Framework Over Time — The Operational Cadence

The framework improves through structured iteration, not through occasional insight. Here is the operational cadence that makes it compound.

W
Weekly — Query Monitoring (20 minutes)
Run 10 priority queries across ChatGPT, Perplexity, Google AI Mode. Record result types. Note new companies appearing. Flag any enriched pages appearing. This is the raw data collection that funds all other improvement cycles. Cannot be skipped.
2W
Bi-Weekly — Evidence Loop Cycle (2 hours)
Run full 20-query baseline. Compare to previous session. Identify field citation movements. Update enrichment prompts for fields that showed lift. Update scoring rubric weights. Add new query variants for types that performed unexpectedly. This is where the prompts improve.
M
Monthly — Normalization Library Update (3 hours)
Process any new suppliers encountered through customer catalog ingestion. Add new supplier quirk profiles. Update color name mapping tables. Validate that existing normalization rules still produce clean output (suppliers change their data formats). This is the infrastructure maintenance cycle.
Q
Quarterly — Framework Publication + Vertical Assessment (1 day)
Compile findings from the quarter into a publishable insight. Share with PPAI Media or industry newsletter. Assess whether the pet supply or ecommerce vertical is ready for entry based on evidence accumulated. Update the scoring rubric with the quarter's empirical data. Review prompt library for structural improvements. This is the external IP deployment cycle.

The One Thing That Makes All of This Defensible

Every step of the framework described above can be replicated by a well-funded competitor given enough time. The normalization library can be rebuilt. The prompts can be reimagined. The measurement methodology can be copied.

The one thing that cannot be replicated is the longitudinal citation dataset — the accumulated record of which fields drove citation lift, for which query types, on which platforms, in which verticals, across which customers, over which time periods.

This dataset does not exist anywhere. No academic institution has it. No competitor has it. No trade association has it. The only way to get it is to run the measurement. Every week you run the 20-query protocol is a week's worth of data that a competitor starting today cannot have.

That is why the query monitoring cadence is the most important operational discipline in the entire framework. Not the prompt engineering. Not the normalization library. The weekly measurement. Because without the measurement, there is no dataset. Without the dataset, there is no defensible IP. Without the IP, the framework is just a methodology that anyone can copy.

Run the queries every week. Record every result. That is how the moat is built.
